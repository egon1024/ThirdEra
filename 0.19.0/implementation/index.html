Implementation overview

This page gives a high-level picture of how ThirdEra is built. It is aimed at anyone extending the system or working with its code, not at end-user usage.

Data models

The system uses Foundry’s DataModel layer for schema and validation. Each actor and item type has a corresponding data model in module/data/:

  • Actor types — Characters and NPCs each have a type-specific data model (e.g. ability scores, level history, spellcasting for characters; stat block, creature type for NPCs). Derived values (ability modifiers, save totals, AC, initiative) are computed in prepareDerivedData() on these models.
  • Item types — Weapons, armor, equipment, spells, feats, skills, races, classes, class features, schools, conditions, creature types, and subtypes are all item types with their own schemas. Booleans are stored as the strings "true" / "false" for Foundry compatibility.

References between documents (e.g. “this feat requires that feat”, “this class grants this feature”) use document ID or UUID, not names or keys, so that compendium and world documents stay unambiguous.

Documents and sheets

  • Actors — One document class (ThirdEraActor) with roll methods (ability checks, saves, skills, initiative, attacks). The sheet is a single ApplicationV2-based sheet that switches templates by actor type (character vs NPC).
  • Items — One document class (ThirdEraItem) with type-specific behavior (e.g. weapons support attack/damage rolls). Each item type has its own sheet template (e.g. weapon, spell, class); the sheet uses Handlebars and submits changes automatically.

Sheets use ApplicationV2 and HandlebarsApplicationMixin; tab switching and actions are handled via data-action and a central actions map. There is no build step: the system is plain ES modules, CSS, and Handlebars templates loaded by Foundry at runtime.

Compendiums

Compendium packs are declared in system.json and populated from JSON under packs/ via a CompendiumLoader that runs when a world is loaded. The loader creates or updates compendium documents so that content can be revised (e.g. name or key changes) without duplicating entries. The source of SRD content (races, classes, feats, spells, etc.) is these compendiums; the system does not hardcode stats. Third-party compendiums or modules can provide additional content using the same item types and structure.

Extensibility

  • Races, classes, skills, and similar are item types, not hardcoded lists. New races or classes can be added via compendiums or world items.
  • Configuration (sizes, damage types, movement maneuverability, etc.) lives in CONFIG.THIRDERA and is used by sheets and logic. The system is designed so that additional packs can extend or override content without forking the core code.

For more detail (entry points, file layout, conventions, and Foundry v13 gotchas), see the Development page on this site; it is the single reference for development and architecture.
